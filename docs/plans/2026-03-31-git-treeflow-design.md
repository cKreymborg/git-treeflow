# git-treeflow Design

A terminal UI for managing git worktrees, launched via `gtf`. Built in Go with Bubbletea.

## Overview

User runs `gtf` in any git repo. A TUI opens showing their worktrees. They can navigate with vim keys, switch to a worktree (which exits the TUI and `cd`s them there), create new ones through a guided flow, or delete existing ones with confirmation.

Single Go binary. Installable via `go install`, Homebrew, or direct download.

## Configuration

Two-layer TOML config. Global defaults at `~/.config/git-treeflow/config.toml`, per-repo overrides at `.git-treeflow.toml` in the repo root. Per-repo values win when both exist.

```toml
# Where to create worktrees relative to the repo
# Default: "../{repoName}.worktrees/{worktreeName}"
worktree_path = "../{repoName}.worktrees/{worktreeName}"

# Files/globs to copy into new worktrees
# Default: [".env*"]
copy_files = [".env", ".env.local", ".env.development"]

# Commands to run after creating a worktree
# Executed in order, from within the new worktree directory
post_create_hooks = [
    "npm install",
    "cp .env.example .env.local",
]
```

### Path Template Variables

- `{repoName}` — name of the git repo directory
- `{worktreeName}` — name chosen during creation
- `{branchName}` — the branch being checked out or created
- `{date}` — current date as `YYYY-MM-DD`

### TUI Settings

The settings view (`s` key from the list) shows the current effective config (merged global + repo). Users can edit values inline. When saving, the TUI asks whether to save to global or repo config.

## TUI Views & Navigation

### Global Keybindings (vim-style)

- `j`/`k` — move up/down
- `g`/`G` — jump to top/bottom
- `q` / `Esc` — quit / go back
- `?` — help overlay showing all keybindings

### 1. Worktree List (default view)

Shows all worktrees: name and branch.

- `Enter` — switch to selected worktree (exits TUI, `cd`s there)
- `c` — open create flow
- `d` — open delete confirmation for selected worktree
- `s` — open settings view
- Active worktree (the one you launched `gtf` from) is highlighted differently

### 2. Create Flow (multi-step)

- Step 1: Enter worktree name (text input)
- Step 2: Choose branch mode — new branch / local branch / remote branch
- Step 3a (new): Enter branch name
- Step 3b (local/remote): Fuzzy-searchable branch list with vim nav
- Step 4: Confirmation summary, then create

After creation, configured `copy_files` are copied and `post_create_hooks` are executed in the new worktree directory.

### 3. Delete Confirmation

- Shows worktree name and branch
- Asks "Also delete the branch?" (yes/no toggle)
- Explicit `y` to confirm, any other key cancels

### 4. Settings View

- Lists current config key-value pairs
- `Enter` to edit a value inline
- `w` to save (prompts: global or repo?)

## Shell Integration

A subprocess can't change the parent shell's directory. Solution: same pattern as `zoxide` and `nvm`.

1. **The binary** (`gtf`) runs the TUI. All TUI rendering goes to stderr (`tea.WithOutput(os.Stderr)`). When the user selects a worktree to switch to, it prints the target path to stdout and exits.
2. **A shell function** wraps the binary, captures stdout, and `cd`s if a path was printed.

```sh
# Generated by `gtf --init zsh` (or bash/fish)
gtf() {
    local result
    result=$(command gtf "$@")
    if [ -n "$result" ] && [ -d "$result" ]; then
        cd "$result"
    fi
}
```

User adds `eval "$(gtf --init zsh)"` to their `.zshrc`.

## Smart Features

### Stash-aware Switching

When switching worktrees, detect uncommitted changes in the current worktree and offer to stash them before switching.

### Direct Jump (`gtf <name>`)

Skip the TUI entirely and jump straight to a worktree by name. `gtf my-feature` `cd`s directly if an exact match exists, fuzzy-matches if not.

### Prune

Detect and offer cleanup for:
- Worktrees whose directories no longer exist on disk (manually deleted)
- Worktrees whose remote tracking branch has been deleted (using `git branch -vv` to check for `gone` status)

Accessible via `p` key from the list view.
